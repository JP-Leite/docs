## Prerequisites

Before you start you need:

- **IAM credentials** — contact your platform administrator to provision a user or service account with at least `CREATE` and `INVOKE` permissions on Agent Builder.
- **API base URL** — provided by your administrator (e.g. `https://api.<ENV>.experience.hyland.com`).
- `curl` or an HTTP client (Postman, Bruno, etc.)

---

## Step 1 — Authenticate

Agent Builder uses OAuth 2.0 client credentials. Exchange your `CLIENT_ID` and `CLIENT_SECRET` for a bearer token.

```bash
curl -X POST "https://auth.iam.<ENV>.experience.hyland.com/idp/connect/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=<CLIENT_ID>" \
  -d "client_secret=<CLIENT_SECRET>" \
  -d "scope=hxp environment_authorization" \
  -d "grant_type=client_credentials"
```

**Response:**

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

Save the `access_token`. Include it as `Authorization: Bearer <token>` in every subsequent request.

<Tip>
Tokens expire after 1 hour. Implement a token refresh mechanism in production.
</Tip>

## Step 2 — Choose Your Access Method

<Tabs>
  <Tab title="Agent Builder Studio (UI)">
    1. Navigate to Agent Builder Studio at `https://agents.<ENV>.experience.hyland.com`
    2. You are redirected to the login screen — enter your IAM credentials
    3. After login, the Studio provides a visual workspace to create agents, test them inline, and manage versions
    4. No token handling required — the Studio manages authentication automatically
  </Tab>
  <Tab title="API / CLI">
    Use the bearer token acquired in Step 1 with any HTTP client. All examples throughout this documentation use `curl`.

    Set a shell variable for convenience:

    ```bash
    TOKEN="<your-access-token>"
    BASE_URL="https://api.<ENV>.experience.hyland.com"
    ```
  </Tab>
</Tabs>

## Step 3 — Create a RAG Agent

A **RAG (Retrieval-Augmented Generation) agent** searches your document repository and answers questions grounded in your content. This is the fastest way to make your Content Lake queryable via natural language.

```bash
curl -X POST "$BASE_URL/v1/agents" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "DocumentHelper",
    "description": "Answers questions from company documents",
    "agentType": "rag",
    "notes": "Initial version",
    "config": {
      "hxqlQuery": "SELECT * FROM SysContent",
      "llmModelId": "amazon.nova-micro-v1:0",
      "inferenceConfig": {
        "maxTokens": 4000
      }
    }
  }'
```

**Response:**

```json
{
  "id": "3f4a12bc-...",
  "name": "DocumentHelper",
  "agentType": "rag",
  "versions": [
    { "id": "v1", "status": "active" }
  ]
}
```

Save the `id` — you need it to invoke the agent.

## Step 4 — Invoke the Agent

Send a question to the agent using its `id` and the `latest` version shortcut:

```bash
curl -X POST "$BASE_URL/v1/agents/<AGENT_ID>/versions/latest/invoke" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "messages": [
      {
        "role": "user",
        "content": "What is our vacation policy?"
      }
    ],
    "hxqlQuery": "SELECT * FROM SysContent"
  }'
```

**Response:**

```json
{
  "object": "response",
  "createdAt": 1741705447,
  "model": "amazon.nova-micro-v1:0",
  "output": [
    {
      "type": "message",
      "status": "completed",
      "role": "assistant",
      "content": [
        {
          "type": "output_text",
          "text": "According to the HR policy, full-time employees receive 15 vacation days per year..."
        }
      ]
    }
  ],
  "customOutputs": {
    "sourceNodes": [
      {
        "docId": "6e4a7f58-...",
        "score": 0.95,
        "text": "VACATION POLICY: Full-time employees are entitled to 15 vacation days..."
      }
    ],
    "ragMode": "normal"
  }
}
```

The `customOutputs.sourceNodes` array shows exactly which document chunks were retrieved and used as context.

## Authentication Methods Reference

| Method | Who uses it | How |
|---|---|---|
| **Login/Studio** | Interactive users | Browser login → session-managed token |
| **Client Credentials (External)** | External services / integrations | Client ID + Secret → JWT bearer token |
| **Client Credentials (Internal/Restricted)** | Internal Hyland services | Terraform-provisioned client → JWT bearer token + environment headers |

For the Internal (restricted scope) flow, include the following headers on every request:

| Header | Description |
|---|---|
| `X-Account-ID` | NCP account identifier |
| `X-Environment-ID` | Target environment |
| `X-User-Context` | User context for the request |

---

## IAM Permissions

| Permission | Allows |
|---|---|
| `READ` | List and retrieve agents |
| `CREATE` | Create new agents and versions |
| `EDIT` | Update agent configuration |
| `INVOKE` | Call agent invocation endpoints |
| `DELETE` | Remove agents |

Request the minimum set of permissions your service needs. If your service only invokes agents, request only `INVOKE` — assign the **Agent Studio Solution Consumers** IAM user group.
