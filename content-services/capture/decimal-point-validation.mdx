---
title: "Decimal Point Validation"
---

Consider the following validation rules for decimal points:

-
Values with more than two characters after the decimal point are flagged as INVALID (for example, 12.456). The reason for this is that during OCR, if the comma (thousands separator) is misread as a period, the incorrect value would be marked as valid. Consider the following examples:

-
If the value on the document is 12,456 and during OCR the comma is misread as a period (12.456), it is getting converted to 12.46 when it should be 12,456.

-
If the value on the document is 12,456,000 and during OCR the last comma is misread as a period (12,456.000), it is getting converted to 12,456.00 when it should be 12,456,000.

-
Values with only one character after the decimal point are flagged as INVALID. For example, if the value on the document is 12,456.71 but OCR captures 12,456.7, converting it to 12,456.70 makes it an invalid assumption.

-
Values with more than one decimal point are flagged as INVALID (for example, 12.456.00).
