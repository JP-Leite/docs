## Overview

The **content model** is the schema system that governs how documents are defined and stored in the CIC repository. It determines what metadata a document carries, how documents relate to each other, and how they can be searched.

Every document in the repository belongs to a **content type**, which acts as a template for its structure.

## Building Blocks

The content model is built from the following core elements:

### Content Type
A **content type** is the set of properties that define a content object. It serves as a blueprint, specifying:
- The list of schemas and metadata fields.
- The document's lifecycle and versioning policy.
- The forms presented to users in the Workspace.

### Property (Metadata)
A **property** is a granular data element stored on a document. Properties can take many forms:
- **Text**: Standard strings.
- **Date**: Calendar dates.
- **Boolean**: Yes/no values.
- **Integers**: Numeric values.
- **Blobs**: Containers for binary content (files).
- **Complex**: Structured data, such as a list of contact objects.

### Mixin
A **mixin** is a reusable data structure that allows properties to be injected into a content type without traditional inheritance. 
- **Efficiency**: Mixins promote reuse and keep your model "DRY" (Don't Repeat Yourself).
- **Dynamic**: A mixin can be added dynamically to a document instance even if the base content type doesn't explicitly reference it.

### Classification
**Classification** provides a way to categorise documents using taxonomy values. This enables dynamic filtering and grouping within the repository, making it easier for users to browse and find content.

## Best Practices

### Using Reusable Mixins
Design your content model using mixins for shared metadata. For example, instead of defining `Reviewer` and `Approval Date` on every individual content type, create a `ReviewMetadata` mixin. This mixin can then be applied across Invoices, Contracts, and Policies, ensuring consistency and making global updates much simpler.
