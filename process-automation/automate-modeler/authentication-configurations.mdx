---
title: "Authentication Configurations"
---

You can create multiple authentication configurations to use various external services in your Studio Modeler project.

To create an authentication configuration:

1.  Do any of the following:
- Click Create > Authentication.
- Click the vertical ellipsis button next to the Authentications drop-down button in the left-hand pane, then select Create.
If you have an existing authentication configuration, you can upload it as a JSON file instead of creating it from scratch. For more information, see Uploading Project Elements.

The Create Authentication window is displayed.

2.  Enter a name in the Authentication name field and a description in the Authentication description field. Providing an authentication description is optional.
3.  Click Create Authentication. The authentication configuration opens as a new tab. It can also be found in the Authentications drop-down menu on the left.
4.  Select one of the authentication types from the drop-down menu:

| Authentication Type | Description |
| --- | --- |
| Basic Authentication | In the header of the HTTP call, an Authorization header is included with a user and a password as a value. |
| Bearer opaque token | In the header of the HTTP call, an Authorization header is included with a value that starts with the word Bearer, followed by an encoded string that the target service can decode to identify the user. If you use this method, you need to include a non-expiring token. |
| OAuth 2.0 Client Credentials grant | The bearer token is created at runtime using OAuth2. You need to set the authorization server, the credentials of a user in OAuth2 (a client application), and the scopes that are going to be retrieved. This is the default setting without any defined Grant type. |
| OAuth 2.0 Custom grant | The bearer token is created in realtime using OAuth2. You need to set the authorization server, the credentials of a user in OAuth2 (a client application), and the scopes that are going to be retrieved. This is for custom settings with a specific Grant type. |
| Simple Mail Transfer Protocol (SMTP) | This authentication can be used to send mail notification from an application with a domain email address, instead of using the default address. This option works with every SMTP mail configuration. To configure SMTP for Microsoft Azure mail account, you need to follow instructions to Create credentials for Simple Mail Transfer Protocol (SMTP) authentication in the Microsoft Azure documentation. The main requirement is to configure the Communication and Email Service Owner Role. To configure SMTP for Google mail account, you need to follow instructions for Sending emails with SMTP server for Gmail in the Google documentation. |

5.  Optionally, deselect Secured so you can enter a user name and password that is used when contacting the API.  CAUTION: It is not recommended to deselect the Secured checkbox. Only proceed if you fully understand the risks and have a valid reason to do so. If this option is deselected, then the following properties can be filled for all types of authentication configuration except for the SMTP one:

| Parameter | Description |
| --- | --- |
| Client id | When using Client Credentials authentication, enter the client ID, for example: Client-1 |
| Client secret | When using Client Credentials authentication, enter the client secret for the API. For example: client.secret |
| Endpoint | When using Client Credentials authentication, enter the endpoint of the server. For example: client.endpoint. The endpoint entered is validated before you can save the authentication. |
| Scope | When using Client Credentials authentication, enter one or more scope values that constitute a part of the API account you want to access. For example: api.account.one. The scope entered is validated before you can save the authentication. |
| Grant type | When using Bearer Token authentication, enter the bearer token for the API. For example: bearer-token.The token needs to be provided in advance, using the client identity set in the Administration Portal. For more information, see the External Applications section of the Content Innovation Cloud Administration Portal documentation. To learn how to retrieve the token from the identity provider, see the Getting Started section of the Hyland Content REST API documentation. Once the external service has the token, it can call the configured URL, adding the Authorization header to the request containing the token. |

The following details can be filled for the Simple Mail Transfer Protocol (SMTP) authentication configuration:

| Parameter | Description |
| --- | --- |
| Host | The address of the server used to send mails given in the mail documentation. |
| Port | Port number given in the mail documentation. |
| Username | Username of the account that will be used to send the notifications. |
| Password | Password of the account that will be used to send the notifications. |
| From address | The address of the sender that will be used to send notifications. |
| From name | Optional: name displayed as the sender. |
| Reply to address | The address that will be used when the recipient of the notification hits the Reply button. |

6.  Click the cloud button to save the authentication configuration.

The configuration is saved and can be used in the project.
